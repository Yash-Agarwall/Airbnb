🧠 What you did

Now:

👉 This is the key line

🔥 How it actually works
1️⃣ In your route file

You wrote:

const router = express.Router();

👉 This creates a mini app (router)

Then:

router.get("/", ...)
router.get("/:id", ...)
⚡ Important Understanding

These routes are NOT absolute

They are relative to where router is mounted

2️⃣ In your main file

From your code:


const listings = require("./routes/listing.js");

app.use("/listings", listings);
🔗 What Express does internally

It combines paths like this:

Router Path	Final URL
/	/listings
/new	/listings/new
/:id	/listings/:id
🧩 Example
In routes/listing.js
router.get("/", ...)
In app.js
app.use("/listings", listings);

👉 Final route becomes:

GET /listings
💡 Think of it like prefixing
app.use("/listings", router);

means:

“Attach /listings in front of every route inside this router”

🔥 Why we do this
❌ Before (messy)
app.get("/listings")
app.get("/listings/:id")
app.post("/listings")
✅ After (clean)
router.get("/")
router.get("/:id")
router.post("/")

👉 Much cleaner, scalable, modular

router = sub-app for a feature

now we study about cookies and how to send them
🍪 What are Web Cookies?

Web cookies are small pieces of data stored in your browser by a website.

👉 They help websites remember information about you

🧠 Simple Example

When you log into a website:

You enter username/password
Server sends a cookie to your browser
Browser stores it

👉 Next time you visit:

Browser sends cookie back
Website says: “Oh, you’re already logged in”
🔁 How Cookies Work (Flow)
Client → sends request
Server → responds + sets cookie
Browser → stores cookie
Future requests → cookie automatically sent
📦 Example Cookie
Set-Cookie: sessionId=abc123; HttpOnly

👉 Stored in browser
👉 Sent with every request

🔥 Common Uses
1. Authentication
Keep user logged in
Store session ID
2. Personalization
Language preference
Theme (dark/light mode)
3. Tracking (Ads/Analytics)
Track user behavior
Used by tools like Google Analytics
⚙️ Types of Cookies
1. Session Cookies
Temporary
Deleted when browser closes
2. Persistent Cookies
Stay for days/months
Used for “Remember me”
3. Secure Cookies
Sent only over HTTPS
4. HttpOnly Cookies
Not accessible via JS
More secure (used for auth)
🔐 Important for YOU (Backend Dev)

In Express, you’ll use cookies for:

res.cookie("token", "abc123");

And read them using:

req.cookies.token

(using middleware like cookie-parser)

⚠️ Security Notes
Never store sensitive data directly in cookies
Use:
HttpOnly
Secure
Prevent attacks like XSS & CSRF

Session: whenever the client interacts with the server, then each interaction is called as a session
Stateless: when the server doesnt store any info about the session.Eg: HTTP

Stateful: when the info is stored. Eg: ftp

Express session: used to make a session of http as Stateful
on the server side: we will provide the user with a session id,
but we are not saving this info as permanent data in the database, we will 
store it in some Temporary storage.
we cant save the complete onfo on the clinet side as cookies have linited size and Security

on the client side: we will save the id using cookies, next time onwards this id will be sent along with the requrest

Client sends request
→ Server checks cookie for session ID
→ Looks up session ID in temporary storage
→ Finds the real data (who you are, your cart, etc.)
→ Responds

A session is the entire conversation between client and server (login → browse → logout)

also: 
res.locals makes variables automatically available to every EJS template without passing them manually in every res.render(). So instead of passing msg in each route, you set it once in a middleware and every template can access it.

js// one middleware, covers all routes
app.use((req, res, next) => {
    res.locals.success = req.flash("success")
    res.locals.error = req.flash("error")
    next()  // ← don't forget this
})
